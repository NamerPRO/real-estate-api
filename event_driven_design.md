# Проектирование Event-Driven архитектуры

## 1. Анализ событий в системе

В текущей системе управления недвижимостью (Real Estate API) выделены следующие ключевые бизнес-процессы, которые порождают события:

### События (Events)
1. **`UserRegistered`** — Пользователь успешно зарегистрирован.
2. **`PropertyCreated`** — Новый объект недвижимости добавлен в систему.
3. **`PropertyStatusChanged`** — Изменен статус объекта (например, с `active` на `sold` или `archived`).
4. **`ViewingScheduled`** — Пользователь записался на просмотр объекта.
5. **`ViewingCancelled`** — Запись на просмотр отменена пользователем или агентом.

### Команды (Commands), инициирующие события
*   `POST /users` -> генерирует `UserRegistered`.
*   `POST /properties` -> генерирует `PropertyCreated`.
*   `PATCH /properties/{id}` -> генерирует `PropertyStatusChanged`.
*   `POST /properties/{id}/viewings` -> генерирует `ViewingScheduled`.
*   `DELETE /properties/{id}/viewings/{viewing_id}` -> генерирует `ViewingCancelled`.

### Подписчики событий (Consumers)
*   **Indexer:** Обновляет поисковый индекс при создании или изменении объекта.
*   **Notification:** Отправляет email/push-уведомления владельцу о новом просмотре или покупателю о подтверждении.
*   **Analytics:** Агрегирует статистику просмотров и продаж.
*   **Cache Invalidator:** Очищает кеш Redis при изменении данных, чтобы не отдавать устаревшую информацию.

В рамках лабораторной работы реализован **Indexer**, который синхронизует данные между Write Database и Read Database. Вместо остальных подписчиков представлены заглушки.

## 2. Проектирование Event-Driven архитектуры

### Компоненты системы

#### Event Producers (Производители)
*   **Real Estate API (C++ Userver):** Основной сервис, обрабатывающий HTTP-запросы. После успешного выполнения команды (записи в Write БД) он публикует событие в брокер сообщений.

#### Возможные Event Consumers (Потребители)
*   **Indexer:** Отдельный асинхронный обработчик, который слушает события изменения недвижимости и обновляет данные в Read Database для быстрого поиска.
*   **Notifier:** Сервис уведомлений. Слушает события просмотров и регистраций.
*   **Analytics:** Сервис аналитики.

В рамках лабораторной работы Notifier и Analytics не реализованы ввиду того, что это не требуется заданием.

### Типы событий и структура
Все события имеют общий заголовок (Header):
*   `event_id`: UUID события.
*   `timestamp`: Время возникновения.
*   `event_type`: Тип события.
*   `payload`: Структура события.

### Поток событий (Event Flow)
1.  Клиент отправляет `POST /properties`.
2.  Real Estate API валидирует данные, сохраняет объект в MongoDB.
3.  Real Estate API публикует событие `PropertyCreated` в RabbitMQ.
4.  **Parallel Processing:**
    *   *Indexer* получает событие и обновляет Read Database.
    *   *Cache Cleaner* получает событие и инвалидирует кеш списков недвижимости.
    *   *Notifier* может отправить приветственное письмо владельцу или уведомление модераторам.

## 3. Проектирование взаимодействия через брокер сообщений

### Выбор брокера: RabbitMQ
Для данного учебного проекта выбран **RabbitMQ**, так как:
*   Требуется гибкая маршрутизация событий разным потребителям (фан-аут).
*   Гарантии доставки "at-least-once" достаточны для задач уведомления и обновления кеша.
*   Проще в настройке и отладке по сравнению с Kafka для небольших объемов данных.

### Формат сообщений
Используется формат **JSON**. Он человекочитаем и легко сериализуется/десериализуется в C++ (через `userver::formats::json`).

### Гарантии доставки
*   **At-least-once (Как минимум один раз):**
    *   Producer подтверждает публикацию сообщения брокеру (publisher confirms).
    *   Consumer отправляет ACK только после успешной обработки события.
    *   В случае сбоя Consumer'а сообщение будет доставлено повторно.

## 4. Применение паттерна CQRS

### Разделение моделей
В системе явно разделяются операции записи (Command) и чтения (Query).

#### Write Model (Команды)
*   **Ответственность:** Обеспечение целостности данных, валидация бизнес-правил.
*   **Хранилище:** PostgreSQL (Users), MongoDB (Properties, Viewings).
*   **Операции:**
    *   `RegisterUser`
    *   `CreateProperty`
    *   `ScheduleViewing`
*   **Результат:** Генерация доменных событий (`UserRegistered`, `PropertyCreated`, etc.).

#### Read Model (Запросы)
*   **Ответственность:** Быстрое предоставление данных для отображения.
*   **Хранилище:**
    *   Денормализованные представления в in-memory хранилище (здесь для простоты в дальнейшем возможен переход на MongoDB).
    *   Redis Cache (для часто запрашиваемых списков).
*   **Операции:**
    *   `GetPropertiesByCity`
    *   `GetUserLogin`
    *   `GetPropertyDetails`

### Синхронизация через события
События являются механизмом синхронизации между Write и Read моделями.
1.  При выполнении команды `CreateProperty` данные сохраняются в основную БД (Write DB).
2.  Публикуется событие `PropertyCreated`.
3.  Consumer обновляет Read Model (например, производит добавление в in-memory хранилище).
4.  Таким образом, Read Model консистентна с Write Model (Eventual Consistency).

## 5. Реализация простого Event-Driven сервиса

Реализация выполнена на языке C++ с использованием библиотеки userver.

### Структура решения

-   `event_producer_component.hpp/cpp` — компонент-продюсер, который публикует события в очередь `real_estate_events`.
-   `event_consumer_component.hpp/cpp` — компонент-консьюмер, подписанный на очередь `real_estate_events`; асинхронно обрабатывает входящие сообщения внутри пула потоков userver.
-   `docker-compose.yml` — оркестрация зависимостей (брокер сообщений, база данных, сервис).
